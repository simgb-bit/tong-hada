# 통 HADA 기능정의서 (v0.2)

## 한 줄 정의

> **통 HADA는 회의록 도구가 아니라, 회의(통)를 AI가 정리하고 조직 데이터로 쌓아주는 플랫폼입니다.**
> 흩어진 회의를 **① 모으고 → ② AI가 정리하고 → ③ 폴더로 보관하고 → ④ 조직 단위로 분석**합니다.

우선순위 표기: **[필수]** = 지금 만든다 · **[다음]** = 그다음 · **[나중]** = 여유 될 때 · 🆕 = 새로 추가 · ✅ = 구현 완료

---

## 1. 핵심 기능 한눈에 보기

| | 기능 | 한 줄 설명 |
| --- | --- | --- |
| 01 | **회의 만들기 · 일정 연동** | 회의를 등록하면 **Teams 일정에 자동 등록** 🆕 |
| 02 | **회의 기록 모으기** | Teams 녹취 · 텍스트 · 메모 · 음성(녹음/파일)으로 기록 수집 |
| 03 | **AI 정리** | 회의 내용을 결정사항·쟁점 등으로 자동 정리 (사람이 검수) |
| 04 | **폴더로 보관** | 회의록을 폴더로 분류·관리 (클로바노트 방식) 🆕 |
| 05 | **분석 · 조직 데이터** | 유형별·조직별 회의 수, 반복 이슈를 한눈에 |

> 가장 큰 차별점은 **03 AI 정리**입니다. 단, AI는 "초안"만 만들고 **최종 확정은 사람**이 합니다. 특히 **담당자는 AI가 임의로 정하지 않습니다.**

## 3. 기능 상세

### 3-1. 회의(통) 만들기

회의를 표준 양식으로 등록합니다.

- **[필수]** 입력 항목: 통명 · 유형 · 일시 · 주관 조직 · 참석자 · 안건 · 관련 자료 · 상태
  - 통명, 주관 조직만 있으면 저장 가능 (나머지는 선택)
  - 유형: `책임자 통 / 주간 통 / 상시 통 / 기타 통`
  - 상태: `예정 / 진행 완료 / 보류`
  - 참석자: **검색해서 추가** ✅ (사원 1000명+ 규모 대응)
    - 이름·사번·조직으로 검색 → 결과에서 클릭해 추가, 선택자는 칩으로 표시·제거
    - 전체 목록을 나열하지 않음(기본 빈 상태), 결과는 최대 20명까지 노출
    - 저장 형식은 **`이름 (조직명)`** — Teams 표기와 동일하게 동명이인 구분 (예: `심규빈 (AX추진 Cell)`)
- **[필수]** 🆕 **Teams 일정 자동 연동**
  - 회의를 등록하면 일시·참석자 기준으로 **Teams(Outlook) 일정이 자동 생성**되고 참석자에게 초대 발송
  - 회의 정보(시간 등)를 수정하면 일정도 자동 업데이트, 회의를 삭제/취소하면 일정도 취소
  - *(현재 Mock → 실제로는 MS Graph API 연동)*
- **[다음]** 정기 회의(주간 통) 반복 생성
- 캘린더 뷰: 월간 달력에서 회의 확인 (통 기록함 안에 통합되어 있음)

### 3-2. 회의 기록 모으기 (입력)

회의 내용을 4가지 방법으로 모읍니다. 여러 번 입력해도 모두 누적 보관됩니다.

| 방법 | 설명 |
| --- | --- |
| **Teams 녹취** | 버튼 한 번으로 Teams 녹취 가져오기 |
| **텍스트** | 직접 타이핑 |
| **음성·메모** | 파일 업로드 또는 **마이크 직접 녹음**(입력 장치 선택·레벨 미터·미리듣기) → **원본(webm) 자동 저장**. 다운로드 시 MP3 변환. 글자 변환(STT)은 선택. **녹음 메모(메모장)**: 녹음하며 자유 메모 → 종료 시 '메모' 1건 저장(메모 기능 통합) |

- **[필수]** 위 4가지 입력 + 입력 이력 목록
- ✅ **입력 상세 보기 + 작성자** — 입력 기록을 클릭하면 전체 내용을 모달로 확인. **누가 입력했는지(작성자)** 함께 표시 (`tong_inputs.created_by`)
- ✅ **입력 소프트 삭제(복구 가능)** — 잘못 입력한 기록 삭제 시 **AI 요약에서 제외**(`tong_inputs.deleted_at`), "삭제된 입력"에서 복구. 동시 삭제는 멱등이라 안전, 편집 권한자만 가능
- ✅ **음원 자동 저장** — 녹음 종료/파일 선택 즉시 음원이 자동 저장됨(별도 버튼 불필요, 유실 방지). **원본(webm/Opus) 그대로 저장**해 용량 최소화, **MP3 변환은 다운로드 시에만**(Opus 가 음성에서 MP3 보다 효율적)
- ✅ **음원 보관 + 90일 자동 삭제** — 음원 원본을 Storage 에 보관하고 90일 경과 시 자동 파기(요약 텍스트는 영구). 첨부 목록에서 **재생/다운로드/삭제**, "자동 삭제 예정/만료" 표시 (#8)
- ✅ **마이크 진단 보조** — 입력 장치 선택·실시간 레벨 미터·무음 경고·저장 전 미리듣기
- ✅ **동시 녹음 방지(실시간 잠금)** — 한 통은 동시에 한 명만 녹음. 다른 사람이 녹음 중이면 "○○님이 녹음 중" 표시 + 시작 차단. Supabase Realtime Presence 기반(탭 닫힘 시 자동 해제)
- **[다음]** 실제 STT·Teams 연동 (현재 STT 는 Mock, 마이크 녹음·저장·다운로드 변환·녹음 잠금은 실제 동작)

### 3-3. AI 정리 (요약)

모은 기록을 AI가 보기 좋게 정리합니다. (항목 간소화 ✅)

- ✅ 항목: **① 한 줄 요약 ② 전체 내용(구조화 정리) ③ 키워드 ④ Comments(참여자 의견)**
- **Comments**: 참여자가 내용을 확인하고 의견을 남김(댓글식). **작성자 표시**, 보기 권한자도 작성 가능, 본인/편집권한자 삭제 (`tong_comments`)
- (구버전의 쟁점·결론·보류·확인필요·후속과제 항목은 '전체 내용'으로 통합)

- **[필수]** AI가 회의 내용을 정리해서 보여줌 (한 줄 요약, 주요 쟁점, 결론 등)
  - ⚠️ **정리 항목 구성은 아직 확정 전** — 어떤 항목으로 정리할지 도메인 검토 후 확정
- **[필수]** 사람이 직접 수정 가능 (AI 초안 → 사람이 확정)
- **[필수]** 담당자가 불명확하면 임의 지정하지 않고 **`확인 필요`**로 표시
- **[다음]** 실제 AI(LLM) 연동 — 현재는 Mock
- **[다음]** 기록을 추가하면 다시 정리(재요약)

### 3-4. 통 기록함 + 폴더 + 공유 ✅

쌓인 회의록을 찾고, **폴더로 정리**하고, **공유**합니다.

- **[필수]** 회의록 목록 (유형·상태·조직·일시 표시) — **한 줄에 하나씩(가로 리스트 행)** 으로 표시
- **[필수]** 검색 + 필터 (유형 / 상태)
  - 유형 필터는 **커스텀 유형 정의(설정)** 를 기준으로 채움 — 통이 없는 유형도 노출, 정렬 순서(`sort_order`) 반영
- **[필수]** 목록 ↔ 캘린더 보기 전환
- **[필수]** ✅ **페이지네이션** — 목록은 한 페이지 **10개**씩 (필터·폴더 전환 시 1페이지로 리셋)

**폴더 기능 ✅ (클로바노트 방식 — 스마트 폴더 + 개인 폴더, 1단계 평면, 다중 분류)**

좌측 폴더 패널이 두 종류로 구성됩니다.

| 구분 | 항목 | 채워지는 방식 | 사용자 생성 |
| --- | --- | --- | --- |
| **스마트 폴더** | `전체` / `내 통` / `공유받은 통` / `휴지통` | 관여·공유·삭제 상태로 자동 분류 | ❌ 고정 |
| **개인 폴더** | 사용자가 만든 폴더 | 통을 직접 담음 (다중 가능) | ✅ 만들기·이름 변경·삭제 |

- `전체`: **내가 관여한 통** = `내 통` ∪ `공유받은 통` (전 조직 공개 아님) / `내 통`: 내가 **진행했거나 참석한** 통 / `공유받은 통`: 남이 나에게 공유한 통 (내가 진행·참석한 건 제외). 세 폴더 모두 휴지통 제외
- **휴지통 ✅**: 삭제 시 휴지통 이동(소프트 삭제) → **복구 / 영구삭제 / 휴지통 비우기(전체)**, 항목 **읽기전용 보기**, **90일 경과 시 자동 영구삭제**(pg_cron, 음원도 함께 정리)
- **다중 분류(태그형)**: 한 통을 **여러 폴더에 동시에** 담을 수 있음 (예: `1분기`와 `OO프로젝트`)
- 분류 진입점 4가지: ① 카드 `⋯` **다중 체크리스트**, ② **드래그앤드롭**(카드 → 좌측 폴더, 데스크톱), ③ **다중 선택 → 일괄 추가/휴지통 이동**, ④ **통 상세·통 생성 화면에서 지정**
- 폴더는 **사람마다 개인 소유** — 공유받은 통을 내 폴더에 담아도 분류는 나만 보임
- **결정 반영**: 하위 폴더 없음(1단계 평면), **개인 폴더만**(조직 공유 폴더 없음), **한 통=여러 폴더 허용(태그형)** → 4번 #2 참고

**공유 기능 ✅**

- 통 상세 화면 `공유` 버튼 → **사원을 검색·선택해 공유**(권한: 보기 / 편집)
- ✅ **참석자 자동 공유** — 통 생성 시 참석자(본인 제외)에게 **편집(입력) 권한 공유가 자동 생성**되어, 참석자의 `공유받은 통` 에 즉시 노출 (회의 등록 → 참석자에게 바로 전달)
- 공유받은 사람의 `공유받은 통` 목록에 해당 통이 나타남
- 공유 대상에서 진행자 본인·현재 사용자·이미 공유된 사람은 자동 제외, 공유 해제 가능
- *현재 권한(보기/편집)은 표시만 — 실제 편집 제어는 인증 도입 시점에 적용*

- **[다음]** 정렬(일시순 등)·기간 필터, 조직 단위 공유

### 3-5. 홈

오늘 할 일과 최근 현황을 한 화면에서 봅니다.

- **[필수]** 오늘 예정된 회의 · 전체 회의 수
- **[필수]** 최근 회의 기록 (최신 5건)
- **[필수]** 반복 이슈 키워드 (자주 나오는 쟁점)

### 3-6. 분석

회의 데이터를 조직 운영 신호로 가공합니다.

- **[필수]** 유형별 / 조직별 회의 수, 반복 키워드, 최근 쟁점
- **[다음]** 기간·조직 필터
- **[나중]** 월별 추이, 특정 키워드 급증 같은 이상 신호

### 3-7. 조직 데이터 (복원 예정) [다음]

조직 계층(Company → CoreGroup → Core → Cell)으로 회의 현황을 봅니다.

- 상위 조직 요약 → 눌러서 하위로 파고들기(drill-down)
- 리더는 자기 조직 + 하위 전체를 볼 수 있음 (보기 전용)

> ⚠️ 현재 메뉴는 제거된 상태. 조직 계층 데이터 자체는 코드에 있어 복원 가능.

| 7 | **개인정보** | 녹취·참석자 정보 보존 기간·접근 통제 (#8 과 연계) |

| 9 | **STT 변환 방식·비용** | ⏳ **상위 논의 필요** — 현재 Mock. 전사(1000명+) 전수 변환 가정 시 비용·개인정보 이슈 (아래 4-2 참고) |

### 4-1. 녹취 음원 보존 (#8 상세)

**현재 동작 (구현 완료):** 녹음/업로드한 음원 원본을 **Storage 에 보관**합니다.
- 녹음 종료(또는 파일 선택) 즉시 **원본(webm/Opus) 자동 저장**(`recordings` 버킷). 별도 버튼 불필요, 유실 없음
- **MP3 변환은 다운로드 시에만** — 저장은 작은 원본으로(Opus 가 음성에서 MP3 보다 2~3배 효율적), 외부 호환은 다운로드 순간 mp3 변환으로 해결
- 저장되는 것: **① 음원 원본 파일**(`storage_path`) + **② 첨부 메타데이터**(`expires_at` 포함) + **③ STT 변환 텍스트**(선택 시)
- 첨부 목록에서 **재생·다운로드(mp3)·삭제** 가능, **90일 경과 시 음원만 자동 파기**(요약 텍스트는 영구)
- 참고: STT(글자 변환)만 아직 Mock, 녹음·저장·재생·다운로드 변환은 실제 동작

**✅ 결정: C안 — 음원을 보관하되 보존기간 경과 시 자동 삭제**

| 방식 | 설명 | 트레이드오프 |
| --- | --- | --- |
| A. 변환만, 음원 폐기 (기존) | 텍스트만 남기고 음원은 저장 안 함 | 저장 비용·개인정보 부담 최소 / 재청취·재변환 불가 |
| B. 음원 보관 | Supabase Storage 등에 원본 보관 | 다시 듣기·재변환 가능 / 스토리지·접근통제·보존기간 정책 필요 |
| **C. 기간 보관 후 자동 삭제 (채택)** | 일정 기간만 보관 후 자동 파기 | 절충안 / 보존기간·자동삭제 운영 필요 |

**구현 메모 (C안) — ✅ 구현 완료**
- **보존기간: 90일** (업로드일 + 90일 = 보관 만료 시각). 요약 텍스트는 영구 보관, **음원만 90일 후 자동 파기**
- 앱: 음원 원본을 Storage(`recordings`)에 업로드(`src/lib/storage.ts`) + `Attachment.expires_at` 기록, 첨부 목록에 "자동 삭제 예정/만료" 표시
- 자동 삭제: `pg_cron` 일일 작업이 만료 음원 → Storage 삭제 → `storage_path` 비움 (`supabase/cron_purge_recordings.sql`)
- 배포 SQL: `migration_audio_retention.sql`(컬럼·버킷·정책) → `cron_purge_recordings.sql`(예약 작업)

### 4-2. STT 변환 방식·비용 (#9 상세) — ⏳ 상위 논의 필요

**쟁점:** 실제 STT 는 거의 모두 유료(사용량 과금) API. 전사(1000명+) 에서 **모든 팀의 모든 회의를 녹음·전수 변환**할 경우 비용·개인정보 이슈가 큼.

**비용 구조 핵심:** STT 비용은 **직원 수가 아니라 변환한 오디오 시간(분)** 에 비례. 회의 1건 = 음원 1개 = 변환 1회 (참석자 수로 곱해지지 않음).

**규모 추산 (1000명, 전수 변환 가정):** 고유 회의 ≈ **월 ~5,000시간** 추정. Teams 비중 **약 20~30%** (사내 추정) → 나머지 **70~80%(~3,500~4,000시간/월)** 가 유료 STT 대상.

| 방식 | 월 비용(약 4,000시간) | 비고 |
| --- | --- | --- |
| Whisper API ($0.36/h) | ~$1,300~1,500 | 운영 부담 0, 단가 최저 |
| Azure Speech ($1/h) | ~$3,500~4,000 | 한국어·화자분리·M365 강점, 고가 |
| **자체 호스팅 Whisper** | ~$500~1,000 **(고정)** | GPU 서버+운영 필요, 양 늘어도 비용 거의 불변 |

**현재까지의 분석/방향 (미확정):**
- Teams 회의분은 **MS Graph 녹취록으로 무료 처리** → 유료 대상 축소 (단, Teams 비중 20~30%로 낮아 효과 제한적)
- 전수 변환 전제라면 **자체 호스팅 Whisper 가 비용·개인정보(사내 보관) 양면에서 유력**. 단 **GPU 서버 + 변환 큐 + 운영 인력** 부담이 트레이드오프
- 민감 데이터(회의 녹취)를 외부 클라우드로 대량 전송하는 것은 **데이터 거버넌스 심의** 필요. 국내 옵션은 Naver CLOVA
- 키는 클라이언트 노출 금지 → 서버(Edge Function 등) 경유

**결정 필요 사항:** ① 전수 변환 vs 선별(중요 회의만) ② 클라우드 API vs 자체 호스팅 ③ 외부 전송 가능 여부(보안 정책) ④ 예산 상한. → **상위 의사결정 후 `src/lib/stt.ts` 교체**

## 5. 데이터 구조 (개념)

```
조직 (Company → CoreGroup → Core → Cell)
 └ 회의(통)  ← 모든 데이터의 중심
     ├ 진행자(created_by)  ← "내 통" 판별 ✅
     ├ 입력 기록 (Teams녹취 / 텍스트 / 메모 / 음성)
     ├ AI 요약 (사람이 수정 가능)
     ├ 첨부파일 (음성 등)
     ├ 공유 (TongShare: 사원 × 권한) ✅
     └ 폴더 분류 (개인 폴더 ← FolderItem 매핑) ✅
```

- 회의록(통)이 데이터의 중심이고, 조직·참석자는 그 위의 "태그"입니다. → 사람이 바뀌거나 조직이 개편돼도 기록은 그대로 보존됩니다.
- **폴더는 개인 소유**(`Folder.owner_id`)이고, 통은 매핑(`FolderItem`)으로 폴더에 담깁니다. → 같은 통을 사람마다 다른 폴더로 분류해도 서로 영향이 없습니다.
- **공유**(`TongShare`)는 통 단위로 특정 사원에게 부여하며, 폴더 분류와는 독립적입니다.
